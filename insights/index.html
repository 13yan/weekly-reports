市场资讯工作台 3 周迭代的关键决策、踩坑记录与反思

2026 年 6 月 搭建周期 3 周 5 个核心功能迭代

序为什么搭一个工作台

市场资讯观察周报是每周末必做的工作。从引力引擎采集数据 → 生成 HTML 报告 → 同步到 GitHub Pages，这套流程已经跑了一年多。

但当我想快速对比"上周和这周"、"看几个关键指标有没有变化"时，每次都要点开完整的 HTML 报告翻找。信息是有的，但获取成本太高。

所以在 5 月底，我决定搭一个工作台 SPA——一个能聚合多周数据、突出关键变化、支持快速浏览的"指挥中心"。

这篇文章记录了搭建过程中的关键决策、踩过的坑、以及一些反思。它不是技术教程，而是一份关于"如何把一个工具打磨成产品"的复盘。

01设计之旅：从微信绿到深邃商务蓝

工作台最初的主色是微信绿#0a7c42，延续了周报本身的品牌色。但跑了几天后，我看着这个绿油油的页面，总觉得哪里不对：

  • 微信绿代表"业务来源"，但工作台要承载的是信息分析，不是游戏发行
  • 绿色在心理上给人"娱乐、轻松"的感觉，与"市场情报、决策参考"的定位错位
  • 顶级咨询公司（彭博、FT、麦肯锡）的产品都是深蓝为主——这不是偶然

我尝试了 3 个备选方向：

深邃商务蓝
#1e3a8a
沉稳、专业
高级灰金
#d4af37 + 黑
奢华、但装饰过重
极致黑白
#1a1a1a + 白
极简、但太冷

最终选了深邃商务蓝。一个细节：图表的色板也跟着调了——主色是 #1e3a8a，第二色用 #60a5fa（浅蓝），第三色是 #f97316（橙），用色温差异建立层次。

设计原则
设计不是"选颜色"，是"选情境"。主色决定整个产品的情绪——同样的数据，放进绿色页面是"业务报告"，放进深蓝页面是"市场情报"。

02布局哲学：留白与密度

我开始的设计稿是 content-max: 1240px，但实际跑起来后问题暴露了：

  • 桌面端屏幕普遍 1440px+ → 两侧各 100px+ 的留白，内容区看着像嵌在空房间里
  • 视觉重心被分散，重要内容反而显得"小"
  • 表格在窄列里数字读着费劲

最终把 content-max 提到 1620px（接近全屏宽度），同时：

  • 卡片圆角从 12px 缩到 10px
  • 卡片内边距从 24px 缩到 20px
  • 表格字号 13.5px（既不显拥挤，也不至于失焦）
小技巧
让内容"撑满"是高端感的关键——留白太多反而显得简陋。一个反直觉的规律：高端感来自"密度刚好"，而不是"刻意留白"。

03Tier 1 新鲜感：让工作台"活"起来

我观察到工作台有"内容齐全"但"打开即静态"的问题——切到不同周，4 个统计数字（微信上榜、抖音上榜、跨平台、品类数量）只是孤零零的"100"。

我心里冒出一个问题：

怎么让工作台跟周报更新节奏绑在一起，感觉更"活"？

我提出 3 层方案：

✅ Tier 1：单周内的新鲜感（已上线）

  • 4 个统计数字后加环比箭头（▲ +5 / ▼ -2）
  • Banner 加倒计时："距下次更新 N 天"
  • 最新周显示红底白字 最新 徽章
  • 老徽章 LIVE 和 历史 都去掉（更干净）

❌ Tier 2：跨周趋势（已回退）

我花了一下午实现：在页面底部加 4 张迷你折线图，展示近 5 周的指标变化。代码上完美工作，但我打开后看着那张图，心想：

自问
"这个数据有用吗？5 周的样本太少，看不出趋势，只是看起来好看。"

直接 git revert 回退。没有数据支撑的"好看"就是噪音。

💡 Tier 3：发布联动（不需要做）

最后发现：工作台每次打开都从 manifest.json 拉所有周报，周五新数据发布后刷新页面就能看到——本身就是"自动同步"了。Tier 3 没有做的必要。

设计原则
克制是新功能的第一原则。每一个新功能都要回答"它在解决什么问题"。答不上来，就别做。

04侧边栏改造：按月收纳

之前侧边栏是 5 条平铺的周报链接。问题：

  • 周报多起来后滚动太长
  • 想找"5 月某一周"得在 5 条里逐一翻

按月分组后：

  • 当前月（6 月）默认展开
  • 历史月（5 月）默认折叠，可点击展开
  • 当月第 1 条（最新周）显示红底白字 最新 徽章

实现上很轻量：从 id.substring(0, 6)（如 "202606"）提取年月，作为分组 key。

05一次失败：Tier 2 趋势概览的回退

值得单独记录这次失败——

我加完 4 张折线图后，心满意足地 push 上线。第二天我自己打开页面，看到 4 张图，愣了一下。

自省
"这图想表达什么？5 周的样本能看出趋势吗？还是只是因为'图看起来高级'所以加了？"

回退这一次教会我：

不要为"存在"而存在。每一个新功能都要回答"它在解决什么问题"。答不上来，就别做。

同样，删除比增加更难，也更重要。回顾这 3 周：

  • 删顶部 topbar（信息冗余）
  • 删 LIVE 徽章（更干净的视觉）
  • 删 历史 徽章（当前月只有一条需要标注）
  • 删 Tier 2 趋势图（数据不足）

每一次删除都让产品更聚焦。

06技术踩坑

6.1 CRLF vs LF 换行符

工作台 HTML 在 Windows 上保存是 \r\n（CRLF），但 Node.js 字符串 '\n' 是 LF。一次替换脚本静默失败，调试了半小时才发现。

症状：Node 脚本输出 "OK" 但文件内容没变。

经验：涉及文件改写的脚本，用正则 \r?\n 兼容两种换行符。

html.replace(/(foo)\r?\n(bar)/, function(m, p1, p2) {
  return p1 + '\r\n' + p2;
});

6.2 删除 UI 元素后的空引用

我删了顶部 topbar（觉得信息冗余），但忘了改 fillData() 里 document.getElementById('currentWeekLabel').textContent = ... 这一行。结果是：

  • Sidebar 正常显示（说明 manifest 加载成功）
  • 主区显示"网络连接失败，请检查网络后重试"（完全误导）

实际原因是 TypeError: Cannot set properties of null（给 null 设 textContent）被外层 .catch() 捕获，显示网络错误。

经验
错误信息要看情境。"网络连接失败"不一定是网络问题，可能是 JS 抛错被外层 catch 误显示。

6.3 sanitizeHtml() 的三轮正则

原 sanitizeHtml 用 3 轮正则处理 HTML：

  1. 去 <script>
  2. 白名单允许 <br>、<strong> 等
  3. 兜底转义所有其他 HTML（防 XSS）

问题：第 3 步会把第 2 步允许的 <br> 又转义回去。多行结论渲染会显示成 "<br>" 文字。

经验：涉及多行结论时，手动按段拼接（<br><br>）比依赖字符串替换更可靠。

07经验总结

1
设计不是"选颜色"，是"选情境"。主色决定整个产品的情绪——同样的数据，放在不同情境里意义完全不同。
2
克制是新功能的第一原则。不为"存在"而存在——每一个新功能都要回答"它在解决什么问题"。
3
删除比增加更难，也更重要。删 topbar、删 LIVE 徽章、删趋势图，每一次删除都让产品更聚焦。
4
错误信息要看情境。"网络连接失败"不一定是网络问题，可能是 JS 抛错被外层 catch 误显示。
5
跨平台部署要测试缓存。GitHub Pages 部署需要 1-2 分钟，CDN 还有 10 分钟缓存期。
6
环境差异要主动兼容。CRLF/LF、file:// 限制、浏览器版本——写脚本时多一层兼容比 debug 三小时便宜。
